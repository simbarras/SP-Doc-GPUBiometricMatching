\section{Analysis}
This project focuses on optimizing the matching component of Miura's pipeline~\cite{MiuraLineTracking2004} to support \textbf{1-to-N} matching. The primary objective is to leverage a \acrshort{gpu} to ensure efficient processing times even when searching against a large database of users.

\subsection{The Miura Pipeline}
The Miura pipeline, introduced in \cite{MiuraLineTracking2004}, is a method that processes a single image of a finger to extract vein patterns. It then computes the distance between these extracted veins and a reference pattern. The method is recognized for its high performance and accuracy, demonstrating the viability of using finger vein patterns as a biometric attribute for authentication.

\subsubsection{Previous Work}
\label{subsubsec:previous-work}
Since its publication in 2004 \cite{MiuraLineTracking2004}, several research projects at the EPFL LASEC lab have built upon the Miura pipeline, specifically exploring its use for patient identification and authentication in hospital environments.

\paragraph*{Optimized Pipeline}
In 2024, Lara Sofie Lenz developed an optimized implementation of the Miura pipeline~\cite{OptimizedPipelineLaraSofieLenz24}. This C++ implementation is approximately 10 times faster than the original Python version for \textbf{1-to-1} matching. The flow of this optimized pipeline is illustrated in Figure~\ref{fig:optimized-pipeline}.

\begin{figure}[ht]
    \mypar

    \caption{The flow of the optimized pipeline, adapted from Lara Sofie Lenz's report~\cite{OptimizedPipelineLaraSofieLenz24}}
    \label{fig:optimized-pipeline}
\end{figure}

This optimized implementation serves as the foundation for this project. While the extraction stage of the pipeline remains unchanged, the matching stage—currently designed for 1-to-1 matching—will be modified to perform 1-to-N matching using a \acrshort{gpu}.

\paragraph*{Mathematical Background}
As shown in Figure~\ref{fig:optimized-pipeline}, the optimized pipeline utilizes a \acrfull{fft} to compute the cross-correlation between extracted veins and the reference pattern. A detailed mathematical explanation of using \acrshort{fft} for signal cross-correlation can be found in Vincent Luke Ventura's project on Biometric Homomorphic Matching~\cite{VincentLukeVentura26}.

\subsubsection{1-to-N Matching}
Previous research at LASEC regarding the biometric identification methods mentioned in Section~\ref{subsubsec:previous-work} has demonstrated strong performance (under 1 second) for \textbf{1-to-1} matching. While 1-to-1 matching is inherently faster, it requires users to provide an ID before authentication. In contrast, 1-to-N matching is essential during the registration phase to ensure a user is not already present in the database.
\mypar
The objective is to compute the cross-correlation between the extracted veins and all reference patterns in the database simultaneously. This requires a shift in architectural design from the \acrshort{cpu}, a general-purpose processor, to the \acrshort{gpu}, which is specialized hardware designed for massive parallel computation.
\mypar
The scope of this project is limited to modifying the matching stage of Lenz's optimized pipeline~\cite{OptimizedPipelineLaraSofieLenz24}. The extraction logic will remain intact, while the matching component will be redesigned to leverage the \acrshort{gpu} for simultaneous cross-correlation across the entire database.

\subsection{Graphics Processing Units}
\acrfull{gpu}s are specialized hardware components designed for efficient parallel computation. They typically function as co-processors to a \acrshort{cpu}, accelerating specific data-parallel applications.

\subsubsection{Architecture}
The standard architecture for a computer's main processor is the \acrshort{cpu}. It is designed for a broad range of tasks and is optimized for low-latency, single-threaded (or few-threaded) performance. It is generally categorized as \acrlong{sisd}, although modern multi-core \acrshort{cpu}s do support levels of parallelism.
\mypar
Conversely, the \acrshort{gpu} is built for high-throughput parallel tasks and is classified as \acrlong{simd} because it can physically execute the same instruction across multiple data points simultaneously.
Note that Nvidia refers to their architecture as \acrlong{simt} to emphasize that while threads execute the same instruction.
\mypar
A \acrshort{gpu} typically consists of several Graphics Processor Cluster (GPC), each containing multiple \acrfull{sm}.
These \acrshort{sm}s contain multiple cores, and each core executes a thread. Threads are logically organized into blocks.
A key distinction of the \acrshort{gpu} is that all threads within a block execute \textbf{the same instruction} at the same time.
\mypar
As illustrated in Figure~\ref{fig:gpu-architecture}, \acrshort{gpu}s utilize a different memory hierarchy than \acrshort{cpu}s, prioritizing high throughput over low latency. The L2 cache is shared across all \acrshort{sm}s, and while global memory is accessible by all threads, it has higher latency compared to the shared memory available to threads within a specific block.

\begin{figure}[ht]
    \mypar
    \centering
    \includegraphics[width=0.85\textwidth]{figures/gpu/GPU-architecture.png}
    \caption{A GPU architecture schema~\cite{vmwareGpuArch}}
    \label{fig:gpu-architecture}
\end{figure}

While different manufacturers have unique thread organizations and memory hierarchies, the underlying principle remains the same. The primary differences usually lie in core counts and memory bandwidth.
Alternative architectures, such as Apple Silicon, employ a unified memory architecture where the \acrshort{cpu} and \acrshort{gpu} share a single memory pool, allowing both to access data without copying it.

\subsubsection{Manufacturers and Programming Models}
In 2026, the two dominant \acrshort{gpu} architecture designers are Nvidia and AMD. Emerging competitors include Intel, with their Intel Arc series, and Apple, which integrates \acrshort{gpu}s into their Silicon chips.
Additionally, Google's \acrfull{tpu} isn't a \acrshort{gpu} but it is a specialized hardware for machine learning.
\mypar
Utilizing these devices requires a programming model to offload data and computations. \acrshort{cuda} is a proprietary model developed by Nvidia and is exclusive to their hardware. AMD's \acrshort{hip} is another programming model designed to allow code to run on both Nvidia and AMD hardware. \acrshort{opencl} is an open standard supporting a wide range of devices from various manufacturers.
\mypar
The choice of programming model is critical for performance and portability. While \acrshort{opencl} offers broad compatibility, its performance is not always optimal, the coding style is dated, and some manufacturers have reduced their support for it.
\mypar
\acrshort{sycl}~\cite{sycl2020} is an abstraction layer that enables code execution across different hardware manufacturers. Unlike \acrshort{opencl}, \acrshort{sycl} provides a high-level, modern C++ style that compiles to the target architecture's native model (such as \acrshort{cuda} or \acrshort{hip}). This is highly beneficial during development, as the specific end-device may not yet be known.
\mypar
Performance remains a key consideration. According to "Comparing Performance and Portability Between CUDA and SYCL for Protein Database Search on NVIDIA, AMD, and Intel GPUs"~\cite{syclVsCuda2023}, \acrshort{sycl} achieves minimal overhead performance compared naive model like \acrshort{cuda} or \acrshort{hip}. 
\mypar
For this project, \acrshort{sycl} will be used to implement the 1-to-N matching. It provides broad compatibility and simplifies development on workstations that may lack a specific \acrshort{gpu} brand.

\subsubsection{GPU Programming}
\label{subsubsec:gpu-programmation}
Since the \acrshort{gpu} acts as a co-processor, code is executed on the \acrshort{cpu} by default. To utilize the \acrshort{gpu}, developers must explicitly offload computations and data. Figure~\ref{fig:gpu-memory} illustrates the memory organization of a system with a \acrshort{gpu}. Data in the host memory (\acrshort{ram}) is not directly accessible by the \acrshort{gpu}; it must be copied to the device memory (\acrshort{vram}) before processing.
\begin{figure}[ht]
    \mypar
    \centering
    \includegraphics[width=\textwidth]{figures/gpu/gpu-memory.png}
    \caption{Memory hierarchy in a PC with a discrete GPU}
    \label{fig:gpu-memory}
\end{figure}
\mypar
The standard workflow for \acrshort{gpu} acceleration, regardless of the programming model, is as follows:
\begin{enumerate}
    \item Allocate variables in the host memory.
    \item Allocate corresponding variables in the device memory.
    \item Copy data from the host memory to the device memory.
    \item Launch the kernel on the device.
    \item Copy the results from the device memory back to the host memory.
    \item Free the device memory.
\end{enumerate}
Appendix~\ref{appendix:sycl-matmul} provides an example of SYCL code for matrix multiplication following these steps.
